# Magick Tool Calling Recommendation Plan

## Intent

Magick should add tool calling as a research and document-workflow feature, not as a terminal-first coding-agent runtime.

The MVP should let the assistant:

- explore the workspace file tree Magick already exposes
- read and write markdown documents inside that workspace
- fetch source material from specific URLs
- show tool activity in a way that supports learning, note-taking, and provenance

The MVP should stay simple:

- small built-in tool set
- no shell or arbitrary process execution
- no MCP
- no plugin/custom tool loading yet
- no deferred tool loading yet
- no parallel tool scheduler yet

The architecture should still make later changes easy:

- richer document-aware tools
- approvals beyond the workspace root
- more providers
- more tool types
- eventual plugin/MCP expansion

## Recommendation Summary

1. Make tool calling an orchestration-level feature owned by Magick, not a Codex-only implementation detail.
2. Keep tool definitions thin and provider-agnostic.
3. Add a small server-side workspace access layer so tool execution can safely operate on the same file tree Magick already uses.
4. Persist tool activity as first-class thread state alongside chat messages instead of faking tool output as assistant prose.
5. Use a minimal permission model: auto-allow reads and writes inside the workspace root, ask only when a tool tries to go beyond it.
6. Give the tools generic names; keep workspace scoping in policy and descriptions rather than in the tool ids.
7. Expose workspace-relative paths to the agent by default so it can navigate only within the workspace without seeing the full absolute path.
8. Show in-chat diffs for file edits and file creation so document changes are inspectable in the thread itself.
9. Present tool activity as compact research actions with expandable details, not as a console transcript.

## Why This Direction

The comparison doc shows two useful lessons:

- Claude Code has the more sophisticated runtime, but its tool model is tightly coupled to Anthropic-specific protocol features and a coding-agent workflow.
- OpenCode has the cleaner and more extensible platform shape: thin tools, a central registry, a simple permission layer, and explicit tool activity in session state.

Magick should borrow the simple platform shape, not the coding-agent product shape.

That means:

- no attempt to copy Claude-style deferred loading or deep tool-specific rendering hooks in the MVP
- no attempt to turn Magick into a shell-oriented agent
- yes to a clean server-owned registry/executor/permission layer
- yes to a UI that emphasizes sources, document actions, and inspectable research steps

## Product Shape For The MVP

Tool calling should be available in Magick chat threads first.

The assistant should be able to roam the workspace root Magick already knows about, read and update markdown files there, and bring back external research material from the web.

For now, the assistant should only see workspace-relative paths. Absolute filesystem paths should stay internal to Magick.

The UI should keep assistant prose and tool activity separate:

- assistant messages remain the place for synthesis, explanations, and writing help
- tool activity rows show what Magick searched, read, fetched, or wrote
- every tool row should be expandable so the user can inspect paths, URLs, previews, and errors

The first release should not try to add document anchors, selection-aware tools, or embedded tool traces inside the editor. Those should be staged after the core tool loop works.

## Initial Tool Set

Ship only these built-ins in the first implementation:

1. `list`
2. `read`
3. `write`
4. `glob`
5. `grep`
6. `apply_patch`
7. `fetch`

### Tool roles

`list`

- list directories and files within the workspace root or any of its sub-directories using workspace-relative paths only
- the main way the model roams the file tree

`read`

- read one markdown file
- accept a workspace-relative path and never expose the absolute root path
- return content plus a concise preview for the UI

`write`

- create or overwrite one markdown file inside the workspace root or any of its sub-directories
- accept a workspace-relative path and keep the actual root path hidden from the model
- support note drafting, editing, and research capture
- return a diff-style preview in the chat UI, including all-added output for newly created files

`glob`

- find files by pattern inside the workspace root or any of its sub-directories
- return workspace-relative matches only
- the main discovery tool for file-path exploration

`grep`

- search file contents inside the workspace root or any of its sub-directories
- report matches with workspace-relative paths only
- keep it text-first and markdown-first in the MVP

`apply_patch`

- make targeted edits to existing files inside the workspace root or any of its sub-directories
- accept only workspace-relative file references in the model-facing interface
- keep patch application centralized so Magick can preview and explain exact edits
- return a diff-style preview in the chat UI for the exact changed hunks

`fetch`

- fetch a URL and produce clean text/markdown for the model and UI
- useful for user-supplied sources without introducing web search yet

### Explicit non-tools for the MVP

- no shell or bash
- no git
- no browser automation
- no LSP
- no image tools
- no general external filesystem access tool
- no web search tool yet
- no generic batch or parallel tool support

The tool definition shape should stay thin. A good MVP shape is:

- `id`
- `description`
- `inputSchemaJson`
- `parse(input)`
- `execute(args, context)`

This avoids introducing a larger plugin system too early while still keeping the tool ABI stable.

### Tech stack notes

Keep the implementation aligned with Magickâ€™s current stack and constraints.

Recommended choices:

- use plain TypeScript functions, classes, and `Promise`/`async` for the tool platform
- keep `Effect` only where the existing provider stream path already depends on it
- use Node built-ins for filesystem and path handling on the backend: `node:fs/promises` and `node:path`
- use the platform `fetch` API for the `fetch` tool
- keep all shared tool/event types in `packages/contracts` and keep that package schema-only
- keep durable state in the existing event store and snapshots rather than inventing a second persistence path
- render tool and diff state through the existing React + Zustand/projector flow in `apps/web`

For file change previews:

- generate unified diff-style previews for `write` and `apply_patch`
- store structured diff preview metadata in tool activity state so the UI does not have to parse raw patch text from scratch
- if a small diff helper dependency is needed, keep it narrow and text-only; do not add a large VCS or editor dependency for the MVP

### 2. Hide absolute workspace paths from the agent by default

The agent should navigate with truncated workspace-relative paths for now.

That means:

- tool inputs should accept relative paths rooted at the workspace
- tool outputs should return relative paths rooted at the workspace
- persisted tool events and UI previews should default to relative paths too
- absolute paths should stay inside the server-only resolution layer

Implement this as a presentation policy, not as a one-off hack.

Recommended shape:

- `pathPresentationPolicy.ts` owns conversion between internal absolute paths and agent-facing relative paths
- `workspacePathPolicy.ts` owns path normalization, containment, and absolute resolution
- all file-oriented tools call both layers in sequence

Add a single toggle now so the behavior can be removed later without changing every tool:

- default mode: `workspace-relative`
- future mode: `absolute`

This can start as a backend config flag and later move into a product setting.

### 3. Add a real workspace access boundary on the server

This is the most important architectural gap in the current codebase.

Today, provider turns run on the server, but workspace file access is mainly handled outside the server by the desktop/web workspace layer.

For tool calling, Magick needs a server-side workspace access service that:

- knows the workspace root for a given `workspaceId`
- can list, read, write, patch, glob, and grep files inside that root
- enforces path containment and markdown-only policy
- keeps absolute paths internal while returning relative paths when truncation mode is enabled
- stays consistent with the local workspace rules already used by the file tree

For the MVP, keep this simple:

- one workspace root per running Magick environment
- map `workspaceId` to that root through a small `WorkspaceAccessService`
- use direct Node filesystem access on the backend
- keep agent-visible paths relative to the workspace root by default

Do not redesign the entire workspace transport stack yet.

### 4. Keep the tool loop in `ThreadOrchestrator`

`ThreadOrchestrator` should remain the owner of the turn lifecycle.

New turn flow:

1. persist user message and `turn.started`
2. start provider turn
3. persist assistant text deltas as they arrive
4. when the provider requests a tool, persist tool-requested state
5. run permission check
6. if allowed, execute the tool and persist tool lifecycle events
7. submit the tool result back to the provider
8. continue the provider turn until final assistant output or failure

Keep the MVP tool loop strictly sequential.

Do not add Claude-style scheduling or parallel batches in the first version.

### 5. Extend the provider contract minimally

Current provider events only model text deltas and failures.

Extend `apps/server/src/providers/providerTypes.ts` with:

- `tool.call.requested`
- `tool.call.cancelled` optional later, not required in the first pass

Extend `ProviderSessionHandle` with a minimal continuation method instead of building a heavier runtime abstraction now:

- `submitToolResult(...)`

That keeps the change small while still enabling a multi-step tool loop.

### 6. Update the Codex adapter, but do not make the tool system Codex-owned

`apps/server/src/providers/codex/codexResponsesClient.ts` and `codexProviderAdapter.ts` should only do provider-specific work:

- expose tool definitions to Codex in the format Codex expects
- parse Codex tool requests from the response stream
- send tool results back into the Codex continuation loop

The rest of the logic should stay in Magickâ€™s tool platform.

Also replace the current coding-agent system prompt in `codexResponsesClient.ts` with a Magick-specific prompt that:

- frames the assistant as a research and writing helper
- explains the limited initial tool set
- tells the model to favor concise provenance-aware use of tools
- avoids coding-agent assumptions
- explains that generic file tools are still scoped to Magick's workspace root
- explains that file paths are presented relative to the workspace root unless a future config disables that behavior

## Event, State, and Protocol Model

### Keep chat messages, add tool activities

Do not redesign the whole thread model into a generic message-parts system yet.

For the MVP, keep the existing `messages` array and add a parallel tool activity stream to `ThreadViewModel`.

Recommended additions in `packages/contracts/src/chat.ts`:

- `ToolActivityView`
- `ToolApprovalView`
- `toolActivities: readonly ToolActivityView[]`
- `pendingToolApproval: ToolApprovalView | null`
- add `awaiting_approval` to `ThreadRuntimeState`

Recommended new domain events:

- `tool.requested`
- `tool.started`
- `tool.completed`
- `tool.failed`
- `tool.approval.requested`
- `tool.approval.resolved`

Event payloads should carry only the data needed for replay and UI inspection:

- `toolCallId`
- `toolName`
- concise argument preview
- concise result preview
- path or URL metadata, with file paths stored in workspace-relative form by default
- structured diff preview metadata for file writes and patches when relevant
- error summary when relevant

Do not persist full tool output blobs in the event stream in the MVP.

The provider still gets the full normalized tool result, but snapshots and UI should store previews and metadata only.

### Add one explicit approval command

Extend `ClientCommand` with a single approval response command such as:

- `tool.approval.respond`

This should carry:

- `threadId`
- `toolCallId`
- `decision`
- optional approval scope like `once` or `session`

### Update websocket payloads only as needed

Keep the existing websocket model centered on domain events and bootstrap snapshots.

That means updating:

- `packages/contracts/src/ws.ts`
- `apps/server/src/transport/wsServer.ts`

Only enough to move the richer thread snapshot and approval response command through the transport.

## Permission Model

The permission model should be intentionally small.

### Default policy

Auto-allow:

- reads inside the workspace root
- writes inside the workspace root
- list/search inside the workspace root

Ask permission:

- any path access outside the workspace root
- any future tool with side effects beyond the workspace

Deny by default:

- malformed paths
- traversal attempts
- unsupported file types for workspace write operations

### Implementation shape

`toolPermissionService.ts` should make central decisions.

`workspacePathPolicy.ts` should:

- normalize requested paths
- resolve them against the workspace root
- reject path escape attempts
- enforce markdown-only operations in workspace tools

`pathPresentationPolicy.ts` should:

- strip the workspace root from agent-visible file paths in default mode
- reject attempts to smuggle absolute paths into tool arguments when relative mode is enabled
- make the future switch to absolute-path mode a single config change instead of a multi-tool refactor

Keep approvals session-scoped in the MVP.

Do not build persistent user-level permission rules yet.

## UI Recommendation

The UI should feel like a research ledger, not a terminal.

### Thread timeline

Render compact activity rows inline with the conversation in `CommentSidebar`:

- `listed notes/research`
- `globbed **/*.md`
- `grepped for neural network`
- `read notes/research/foo.md`
- `updated notes/draft.md`
- `patched notes/draft.md`
- `fetched https://...`

Each row should show:

- a short label
- completion state
- workspace-relative path or URL when relevant
- a one-line preview

Each row should also be expandable to show:

- input summary
- output preview
- unified diff preview for file edits or new files when relevant
- errors

### Approval UI

When a tool tries to leave the workspace root, show a compact inline blocker card in the active thread with:

- what Magick wants to access
- why it needs approval
- `Allow once`
- `Allow for this session`
- `Reject`

### Important UX guardrails

- keep completed tool rows visually quieter than assistant prose
- default to collapsed detail after completion
- emphasize provenance over raw JSON
- make file diffs easy to inspect without turning the thread into a code review UI
- never dump large raw results inline by default
- keep the assistantâ€™s final answer focused on synthesis and next steps

## Risks And Tradeoffs

### Main risks

- the current provider contract is too small for multi-step tool turns, so the first contract extension needs to be carefully tested
- workspace access now lives in a different part of the stack than provider turns, so root resolution and file consistency need a clean seam
- `fetch` still needs a reliable content-extraction path so fetched pages remain readable and bounded
- path truncation can become scattered and fragile if it is baked into each tool instead of a single presentation layer
- diff previews can become noisy if they are not bounded and collapsed carefully
- approval state introduces a new paused-turn lifecycle that is not represented in the current runtime state enum

### Chosen tradeoffs

- additive `toolActivities` are simpler than a full transcript/message-parts redesign right now
- generic tool ids are cleaner than `workspace_*` names because scope belongs in policy, not naming
- path truncation should be treated as a togglable presentation policy, not a permanent filesystem constraint
- sequential tool execution is simpler and fits the initial research workflow
- a static registry is simpler than early plugin or MCP support

## Explicit Non-Goals For The MVP

- turning Magick into a shell-oriented coding agent
- Claude-style deferred tool loading
- OpenCode-style custom local tools and plugin hooks
- MCP support
- parallel tool execution
- document anchors and selection-aware tool semantics
- full blob persistence of tool outputs
- browser-only fallback support for the full tool loop

The browser-only fallback can stay out of scope until the real desktop/dev-backed server tool flow is stable.

## Final Recommendation

Build Magick tool calling as a small server-owned orchestration layer with a static built-in registry, a workspace-rooted filesystem boundary, additive tool activity state, and a research-oriented UI.

This gives Magick a strong MVP for learning, note-taking, and research without inheriting the complexity or product assumptions of a coding-agent runtime.
